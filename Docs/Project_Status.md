# GoblinFarmer Project Status

This file is the source of truth for current route logic, stable behavior, active work, known issues, recent fixes, and the next recommended task.

## Current Focus
v1.4.0 foundation work now includes stricter VS Debug config-path persistence, startup validation, broader Diablo III auto-discovery for custom drive installs, success screenshot/package default hardening, Witch Doctor mouse-wheel plus cursor-change left-click combat input, a local-only source sync helper for ChatGPT uploads, and the next Goblin Tracker validation fixes. Visual Studio Debug resolves `Config\AppSettings.json` from the project root when available, relative runtime paths resolve from the active config root, config auto-create/migration saves preserve existing non-empty Diablo III, Battle.net, and custom Images values, and missing/blank/invalid Diablo III can no longer produce final `App Status: Idle`. Startup now runs bounded runtime auto-discovery before setup gating, saves a discovered Diablo path to the active config, searches fixed local drives for common Diablo III install roots such as `D:\Diablo III`, prefers `Diablo III64.exe` before `Diablo III.exe`, and never selects `Diablo III Launcher.exe`. Success screenshots remain disabled by default in Release, installed, and VS Debug runs; Debug Mode, Keep Debug Screenshots, and VS Debug evidence defaults do not enable them, and debug packages exclude success evidence unless `-IncludeSuccessScreenshots` is explicitly supplied. Witch Doctor combat no longer uses held-left/channel mouse mode or held-right mode; Witch Doctor startup now launches the `2,3,1` key loop, dedicated mouse-wheel loop, and dedicated cursor-change left-click loop concurrently. The cursor loop sends discrete LEFTDOWN/LEFTUP pulses only through the cursor-change gate and logs startup, cursor checks, sent clicks, skipped clicks, cursor handles, held-state flags, and no-click region state with `combatInputMode=MouseWheelScroll` diagnostics. Startup and validation logs report the active config path, raw loaded Diablo path, whether it was blank, each Diablo candidate root checked, whether auto-discovery ran/found Diablo, the final selected path, and the final validation result. The source sync helper resolves the project root relative to its batch-file location, writes timestamped ZIPs to local ignored `ChatGPT Uploads\`, and stays ignored under `Scripts\Local Tools\` unless explicitly allowed later. The location-aware Goblin Tracker duplicate guard remains implemented and stable with default one-count-per-area behavior plus explicit two-count-per-game overrides for Pandemonium Fortress Level 1 and Pandemonium Fortress Level 2. Manual `X` tracker area resolution now detects very close known-confusable title-template pairs involving Pandemonium Fortress, channels, Caverns of Frost, Moon Clan caves, and Cathedral levels; it uses current raw route context to select the real subarea when appropriate or suppresses unresolved ambiguity instead of consuming the wrong PF area limit. Manual `X` also has an explicit Goblin Tracker no-count block list for WhimsyDale, City of Caldeum, Gates of Caldeum, Caldeum Bazaar, Flooded Causeway, Ancient Waterway, and The Bridge Of Korsikk without consuming area-count slots or changing Teleport Next route blocking. The first Debug Manager / Diagnostics Manager pass remains implemented through `DebugManager` and `DiagnosticsSessionState`, centralizing debug profile behavior, screenshot/session artifact paths, diagnostic throttling, reliability counters, tracker stats, counted-area summary metadata, and active-time snapshots. Reliability work around Battle.net launch, Diablo launch detection, Start Game stability/recovery, Make New Game workflow protection, teleport routing accuracy, combat safety, town workflows, and synchronized app/installer versioning remains stable. Remaining work is live validation and future hardening, not blocking implementation.

Automation Observation Mode is active for `JournalCandidate` and `MinimapCandidate`. Evidence candidates use the existing area resolver, no-count block list, duplicate guard, and PF1/PF2 two-count exception logic to log what would count, but automatic counting is not enabled. Observations do not change GoblinCount, GPH, tracker active time, found records, or counted-area slots. The evidence scanner now starts with the combat token and logs scanner start/stop, scan attempts/skips, candidate found/not-found summaries, and throttled Journal/Minimap diagnostic crop paths under `Debug\GoblinEvidence\ObservationDiagnostics`. Observation templates are discovered from per-goblin PNGs under `Images\Goblin Evidence` using `<Goblin Type> Engaged Journal.png`, `<Goblin Type> Killed Journal.png`, `<Goblin Type> Engaged & Killed Journal.png`, and `<Goblin Type> Minimap.png`. Journal templates remain the primary goblin-type evidence when both Journal and Minimap match in the same scan; tight minimap icon templates are treated as supporting goblin-presence evidence unless no Journal match is available. Missing or invalid template setup logs one clear `GoblinEvidenceTemplateSetupMissing` / `GoblinEvidenceTemplateSetupWarning` line plus throttled `GoblinEvidenceScanResult reason=MissingTemplate` summaries instead of spamming every scan. Journal observation scans use the calibrated `64,736,645,417` reference region, and minimap observation scans use the calibrated `2108,66,421,423` reference region. Candidate checks now log `templateName`, `goblinType`, `source=Journal|Minimap`, `scanRegion`, `bestConfidence`, `threshold`, `matchPoint`, `screenMatchPoint`, and template size. Diagnostic crops continue as reference material only; the app does not auto-create templates from random crops.

VS Debug UI sizing now leaves the expanded Goblin Tracker group fully visible after the Last Observation fields are added. Release-user launches remain compact and diagnostics remain hidden until Debug Mode is enabled.

## Official Route Logic
- Southern Highlands: next Northern Highlands; no block.
- Northern Highlands: next The Weeping Hollow; no block.
- The Weeping Hollow: next The Festering Woods; no block.
- The Festering Woods: next Cathedral; no block.
- Cathedral: next Royal Crypts; block all Cathedral locations except Cathedral Level 3.
- Royal Crypts: next City Of Caldeum; no block.
- City Of Caldeum: next Ancient Waterway; block all City Of Caldeum sublocations except Ruined Cistern.
- Ancient Waterway: next Stinging Winds; Western Channel Level 1 blocks; Western Channel Level 2 allows teleport back to Ancient Waterway; Eastern Channel Level 1 blocks; Eastern Channel Level 2 allows Stinging Winds; manual Ancient Waterway button click while already inside Ancient Waterway blocks.
- Stinging Winds: next Battlefields; Stinging Winds blocks; Black Canyon Mines allows Battlefields.
- Battlefields: next Rakkis Crossing; Caverns of Frost Level 1 blocks Teleport Next hotkey routing; Caverns of Frost Level 2 allows Teleport Next to Rakkis Crossing.
- Rakkis Crossing: next Pandemonium Fortress Level 1; no block.
- Pandemonium Fortress Level 1: next Pandemonium Fortress Level 2; no block.
- Pandemonium Fortress Level 2: next Make New Game flow; no block.
- WhimsyDale is not part of the farming route; if the fresh Teleport Next hotkey current-location scan detects WhimsyDale, Teleport Next is blocked and the notification uses WhimsyDale as the displayed location.
- Cave Of The Moon Clan Level 1 is not part of the farming route; if the fresh Teleport Next hotkey current-location scan detects Cave Of The Moon Clan Level 1, Teleport Next is blocked and the notification uses Cave Of The Moon Clan Level 1 as the displayed location.
- Cave Of The Moon Clan Level 2 is an allowed Southern Highlands route sublocation; if the fresh Teleport Next hotkey current-location scan detects Cave Of The Moon Clan Level 2 while Northern Highlands is queued, Teleport Next should continue to Northern Highlands.
- Caverns of Frost Level 1 blocks Teleport Next hotkey routing and uses Caverns of Frost Level 1 as the displayed location; Caverns of Frost Level 2 remains allowed and can continue to Rakkis Crossing.

### Battle.net Launch Diagnostics
- Launch state is reset at Battle.net launch start.
- Launch startup readiness is based on a visible Battle.net window, not the background `Battle.net.exe` process. Process status is retained in logs only as `processObserved`.
- Launch startup logs configured/detected executable path selection, each launch attempt, retry count, elapsed time, and whether a launch-ready window was detected.
- Play button scanning logs each 100ms polling attempt and sends the click only from the current confidence-passing image match.
- Post-launch close status is based on whether the visible Battle.net window remains. Background/tray Battle.net processes may remain after Diablo launches and are logged as informational, not failures.
- `BattleNetPlayClickSentByApp` logs the app click point and timestamp when GoblinFarmer sends the Play click.
- `BattleNetManualPlaySuspected` logs when Diablo appears during launch flow without a recorded app Play click.
- `BattleNetLaunchSummary` records whether launch should be considered fully automated. It is fully automated only when the app recorded the Play click and Diablo launched afterward.
- `BattleNetStillOpenAfterDiabloLaunch` logs and captures evidence when the visible Battle.net window remains open after Diablo launches, then requests the existing safe Battle.net close path again.

### Runtime Input Cleanup
- `ForceReleaseAllRuntimeInputs` logs cleanup reason, tracked held left/right/Shift state before cleanup, Diablo window handle, and Diablo rect availability.
- Cleanup sends `LEFTUP`, `RIGHTUP`, or Shift key-up only when runtime state says that input is currently held and the Diablo window rectangle is available.
- When Diablo is unavailable, cleanup clears tracked held-input state without generating mouse or key events.
- Combat and hotkey runtime click paths update tracked held-input state, including Demon Hunter right-hold cleanup, Shift-left clicks, loot-click pulses, Kadala right-click pulses, and automation safe clicks.

### Combat No-Click Regions
- Combat no-click regions are Diablo-window-relative rectangles, not absolute screen coordinates.
- `ExtendedRightMenuNoClickRegion`: reference-space rectangle `2410,1120,150,240`, based on the green-box hover menu reference image.
- This region is combat-only and does not affect teleport, repair, salvage, Battle.net launch, Start Game, or Exit Game flows.
- Hover menus are expected in-game behavior; combat should continue running while mouse clicks in the unsafe area are suppressed.
- Current combat mouse input uses physical cursor Win32 `mouse_event`, not `SendInput`, `PostMessage`, or `SendMessage`.
- Window-targeted mouse input has not been implemented because Diablo may ignore or inconsistently handle posted mouse messages, and unsafe coordinates could still interact with UI.
- Safe mode is `PhysicalCursorNoClickSuppression`: suppress new mouse clicks in no-click regions while continuing non-mouse combat actions where possible.
- Demon Hunter right-hold uses `PhysicalCursorHeldFromSafeRegion`: the hold must start in a safe region, then no new right-click is sent while hovering over protected UI.
- Demon Hunter-only `DemonHunterRightHeldNoClickSuppressionActive` means right mouse is already held, shared left-click cursor-loop input is suppressed, keyboard/timers continue, and right mouse is not released.
- Witch Doctor mouse-wheel mode does not hold left or right mouse. Wheel scrolling is logged as `MouseWheelScroll`; cursor-change left-click pulses are discrete LEFTDOWN/LEFTUP events, and unsafe/no-click cursor regions log skipped left-clicks while wheel scrolling and keyboard timers continue.
- Combat keyboard hook suppresses physical `1`/`2` while combat is active or stopping, but allows `LLKHF_INJECTED` key events so automation-generated combat keys are not filtered.
