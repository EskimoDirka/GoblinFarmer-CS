# GoblinFarmer Project Status

This file is the current source of truth for the active release, stable behavior, route/blocking rules, and next development plans. Historical notes and older package reviews live in `Docs/History.md`. Open work and validation items live in `Docs/TODO.md`.

## Current Release

- Version: `v1.5.0`
- Current focus: v1.5 release readiness for salvage reliability, log-first replay evidence, and gem auto-stashing, with ongoing Goblin Tracker validation.
- Automatic counting: implemented behind `GoblinTracker.EnableObservationMode=true` and `GoblinTracker.EnableAutomaticCounting=true`.
- Manual count hotkey: physical `X` has been retired. Goblin counts should come from automatic eligible evidence.
- VS Debug Goblin Tracker controls: `Observation Mode`, `Auto Goblin Count`, `Enable Goblin Decision Trace`, `Capture`, a debug-only simulation area selector, and `Sim Count`.
- VS Debug `Capture` button: manual image-recognition aid only. It writes fullscreen, minimap, journal, and metadata files under `Debug\GoblinEvidence\ManualCaptures` only when clicked.
- VS Debug `Sim Count` button: developer-only count-policy simulator. Its dropdown now uses a centralized route/policy area list built from `GoblinAreaResolver.KnownAreas`, with `Current Area` pinned first and the remaining route/count/block validation areas alphabetized. It uses existing area-key resolution, blocked-area checks, duplicate guard, and area-limit logic so accepted counts, duplicate suppression, and PF1/PF2/Stinging Winds `AreaLimitReached` behavior can be tested without waiting for rare live spawns.
- Automatic debug artifacts: accepted Goblin Tracker count workflows still automatically write decision bundles and encounter captures needed for debugging.
- `Next Tests` tab: removed. Current validation steps are tracked in `Docs/TODO.md`.
- Latest change: normal Goblin Evidence scanner events no longer write redundant root `GoblinEvidence_*` full/event images by default. Decision bundles and encounter captures now carry the replay-ready Journal/Minimap crops, metadata, JSONL, and trace data needed for debugging; the explicit VS Debug `Capture` button remains the only path that intentionally saves a fullscreen image for image-recognition troubleshooting.
- Latest detection policy change: automatic counting still prefers confirmed evidence. Minimap evidence remains eligible after the confidence gate, fresh Journal Killed evidence remains eligible, and first-seen Journal Engaged / combined Engaged-first evidence reports `JournalPendingKilledOrMinimapConfirmation`. If the same Engaged evidence remains in the same area during active combat for the short sustained-confirmation window, it may count as `JournalEngagedSustainedActiveCombat` so real goblins that never produce usable Killed/Minimap frames are not missed.
- Latest utility cleanup: normal tracked package launcher remains `Create Debug Package.bat`. Required backing scripts are `create-debug-package.ps1` and `debug-analysis-tools.ps1`; the debug package script directly loads the helper. `Cleanup Project.bat`, `Cleanup Project Delete.bat`, and `cleanup-project.ps1` are documented maintenance-only exceptions for generated artifact cleanup; the dry-run launcher and PowerShell script default to dry-run, and the delete launcher asks for `Y/N` before passing `-Delete`. The Project Brain ZIP generator (`Create Project Brain.bat` and `create-project-brain.ps1`) was removed after dependency review confirmed it was not used by runtime automation, debug package creation, cleanup, or build/publish code. `Docs\ProjectBrain` remains as direct Codex context documentation.
- Latest agent-docs change: refined the lightweight DOX-style hierarchy to keep `AGENTS.md` minimal, added `Docs\Agent\Invariants.md` for long-lived rules that require deliberate review before changing, and added `Docs\Agent\Task_Checklists.md` for start/completion hygiene. Project Brain markdown docs remain available directly under `Docs\ProjectBrain`; no Agent Zero install, framework, Skill, MCP integration, or app logic change was introduced.
- Latest salvage workflow change: `Repair.EnableBulkCategorySalvage` now controls a blue/yellow-only bulk salvage pass. Release/default config keeps it disabled; VS Debug defaults enable it for validation. When enabled, salvage samples active button color before clicking `Salvage All Blue` or `Salvage All Yellow`, confirms the category prompt with Enter, verifies the prompt clears, then rescans leftovers with the shared salvage inventory classifier. White-item bulk salvage is intentionally absent. Normal gems (`Emerald`, `Ruby`, `Amethyst`, `Topaz`, `Diamond`) are classified as `RegularGemNonSalvageable` and skipped, including stacked one-tile gems, tooltip-covered live gem-stack clusters, amber topaz/ruby-like gem bodies, and weak lower gem cells that would otherwise look like legal two-row footprints; all remaining non-gem targets, including set/legendary-style items, use the confirmation-aware individual salvage path. Salvage item geometry is now explicit: each target footprint must be one column wide and one or two inventory rows tall. Weak upper cells attached to lower green/set pixels remain one two-row target unless the upper cell has its own convincing item boundary. The shared repair-flow salvage path now sets the app status to `Salvaging Inventory` instead of leaving the UI on `Repairing Gear`. If an expected set/legendary confirmation is missed, salvage verifies the cached target with a fresh classifier scan before failing; stale cached targets that are already gone are skipped and handed to the normal final leftover/recovery path. If the final post-salvage scan finds accepted non-gem leftovers, salvage performs up to two bounded recovery rescans/click passes before failing visibly; rejected/unknown leftover candidates remain diagnostic-only.
- Latest stash workflow implementation: `Stash.EnableAutoGemStash` now controls a gem-specific auto-stash step after successful repair/salvage. Release/default config keeps it disabled; VS Debug defaults enable it for validation. After successful salvage, the town flow keeps the existing single Escape close and logs `TownUiClosedForStash escapePresses=1`; it does not send a second Escape that could open the pause menu. Stash assets are separated under `Images\Gems`: gem PNG templates live directly in that folder, and `Gem Coordinates.txt` supplies `Stash Coordinates` plus `Gem Stash Tab Coordinates`. Missing `Images\Gems`, missing coordinates, or no gem templates log a safe skip. When enabled and configured, the workflow clicks the configured stash coordinate as the movement/travel click, waits `Stash.TravelToStashWaitMs` (`1500` ms by default after live timing review), clicks the same stash point to open it, focuses the gem tab after a shorter settle, scans the shared inventory grid once with the production gem-template classifier, right-clicks only accepted gem-template matches, then performs one bounded recovery rescan before logging `AutoStashGemLeftoversRemain` diagnostically and continuing route flow. Debug Mode records stash replay metadata to structured JSONL under `Debug\ReplayLogs`; it no longer writes automatic stash replay screenshot folders.
- Latest debug package evidence change: runtime salvage and gem-stash replay evidence is log-first. Live scans append structured inventory replay events under `Debug\ReplayLogs\inventory-replay-events.jsonl` instead of saving automatic `inventory-grid.png` replay folders. The default `Scripts\Create Debug Package.bat` path now attempts OBS/log alignment automatically: `create-debug-package.ps1` finds the newest video under `Video Clip Review`, parses high-value events from the latest log, estimates video offsets, extracts selected frames with ffmpeg when available, and writes `ReviewEvidence\manifest.json` plus `ReviewEvidence\issue.md` into the normal debug package ZIP. Manual review inputs (`-ReviewVideoPath`, `-ReviewTimestamp`, `-ReviewNotesPath`, `-ReviewEvidenceFolder`, `-MaxReviewEvidenceFrames`) remain available to fill gaps or force specific frames. Full OBS videos remain excluded by default, old `Debug\InventoryReplay` folders are no longer included in debug packages, and explicit package replay uses `--debug-package-replay` against a package ZIP/folder.
- Latest debug package script change: the final `Scripts\create-debug-package.ps1` summary now reports the selected `Video Clip Review` recording path plus duration, file size, and modified time so package creation output identifies the matching OBS clip before the PowerShell window is closed.
- Latest VS Debug UI change: the form now shows an `OBS` status group under `Settings` with recording `Start Time`, `Status` (`recording`, `not recording`, `starting`, `stopping`, or `closed`), and recording `End Time`. The display reads the local OBS auto-record log plus process state and does not change the stable recording script behavior.

Older bundles that contain only `decision-trace.txt` and fullscreen `evidence.png` remain manual-review evidence but are not crop-replayable unless matching capture frames exist elsewhere.
- Goblin Replay/policy coverage is strong enough to consider these validation paths covered unless a future live package contradicts them: Make New Game visible-row carryover suppression, Stinging Winds -> Black Canyon Mines stale-Journal suppression, Black Canyon Mines -> Rakkis Crossing stale-Journal suppression, Southern Highlands -> Cave Of The Moon Clan stale-Journal suppression, Cave/Caverns Level 1 -> Level 2 stale freshness protection, Battlefields journal-history row suppression, same-area duplicate suppression, and default/two-count area-limit policy.
- Goblin Replay does not validate live scanner latency, notification splash timing, Last Observation UI rendering/persistence, real Diablo focus/timing, route hotkey behavior, or package-size retention. Those still need normal VS Debug use/package review.
- Normal scanner evidence events record type/confidence/source and feed observation/auto-count, but skip the old root screenshot file. Replay/debug imagery should come from DecisionBundles, EncounterCaptures, ObservationDiagnostics, or the explicit manual `Capture` button.
- VS Debug `Sim Count` is not a replay feature and is not wired into live scanning. It is an explicit developer button that records through the existing session count path only when clicked.
- VS Debug `Sim Count` area options are centralized in `GoblinTrackerDebugSimulationAreas`. The list includes route count areas, special two-count areas, and blocked/no-count validation areas; the result still depends on existing block-list and duplicate-guard policy rather than a separate simulation mechanism.

- Accepted-count notifications are no-activate and click-through. Rainbow Goblin automatic counts use a distinct alert message plus a local Windows exclamation sound.

- Reset Stats clears count, GPH/active time, duplicate guard, auto-count evidence state, observation state, and Last Observation.
- New Game clears count, GPH/active time, per-game duplicate/evidence state, observation state, and Last Observation so the next game starts clean.

- Legacy Salvage Inventory Replay artifacts: 7-day age retention remains for old matching `Debug\InventoryReplay\Salvage\SalvageInventoryReplay_*` folders so previous runtime artifacts age out.
- Debug package export defaults: `DebugPackageRetentionCount=20`, `MaxGoblinObservationDiagnosticCrops=6`, `MaxGoblinEvidenceFullImages=0`, `MaxGoblinEvidenceEventScreenshots=3`, `MaxGoblinEvidenceEventScreenshotBytes=1048576`, `MaxReviewEvidenceFrames=12`.

Cleanup triggers and behavior:

- Startup runs `DebugManager.CleanupDebugArtifactsByAge`, `CleanupOldSessionSummaries`, `CleanupOldDebugPackages`, and `CleanupOldGoblinEvidence`.
- Age cleanup targets `Logs`, `Screenshots`, `debug-screenshots`, `Sessions`, `DebugPackages`, `Debug\GoblinEvidence`, `Debug\ReplayLogs`, and legacy `Debug\InventoryReplay` under both the runtime base directory and resolved project/config root when available.
- Session summary count cleanup keeps newest matching `Sessions\Session_*.md` files.
- Debug package count cleanup keeps newest matching `DebugPackages\GoblinFarmer_Debug_*.zip` files; startup scans discovered `DebugPackages` folders and `Scripts\create-debug-package.ps1` also prunes the resolved export folder immediately after a batch-created ZIP is written.
- GoblinEvidence count cleanup keeps newest loose files recursively under `Debug\GoblinEvidence`; it is called at startup and after calibration snapshot completion. ObservationDiagnostics also runs its own 24-file cleanup during scanner passes.
- DecisionBundles, EncounterCaptures, ManualCaptures, and replay-ready crops are included in the GoblinEvidence tree and are therefore covered by the startup age/count cleanup. Debug package ZIP contents are additionally bounded by package-script include/exclude limits.
- Salvage/gem-stash replay logs are generated only by debug-mode inventory scans and are included in debug packages as structured JSONL. Runtime no longer creates automatic inventory replay screenshot folders, package creation excludes legacy `Debug\InventoryReplay` image folders by default, and package creation still never invokes replay.
- Goblin Replay template scenarios create only temporary crop frames and delete their temp root at the end of explicit harness execution; Replay is not a normal startup, scanner, package, or form-close cleanup trigger.
